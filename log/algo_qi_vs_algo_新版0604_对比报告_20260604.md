## 4. 重点差异分析

## 4.1 高优先级：闭环仿真 / 批处理 / 总流程分叉

### 4.1.1 `run_full_project_pipeline.py`

当前项目相对于 `algo-新版0604`，主要有下面这些变化：

- 删除了对 `failure_and_attribution_analysis/config/scenario_exploration.yaml` 的依赖。
- 删除了 `--exploration-config`、`--single-attack-types`、`threshold split`、`rolling drift` 等一批 CLI 参数。
- 不再从 YAML 中解析 `single_attack_types`，批处理阶段也不再把该能力继续向下游传递。
- 新增 `latest_coverage`、`failure_detection_accuracy` 两列，并从 `output_summary.txt` 回填到评估结果表。
- 默认仿真命令中固定加入：
  - `--threshold-objective accuracy`
  - `--enable-accuracy-guard`
  - `--min-failure-detection-accuracy`
- `env_config` 解析逻辑改为更偏向“字典配置输入”，并在未使用 `env_md` 时回退到 `DEFAULT_ENV_CONFIG`。
- `run_id` 生成逻辑支持直接复用 `database_writer.task_id`。

结论：

- 当前项目把“探索配置 YAML 驱动的灵活性”收缩掉了。
- 当前项目同时增强了结果追踪，新增了覆盖率/检测精度指标回填，以及精度守卫链路。
- 合并时这里是最高优先级冲突点，需要先决定“保留配置驱动”还是“保留当前简化版命令链路”。

### 4.1.2 `iterative_testing/iterative_failure_simulation.py`

这是本次分叉中差异最大的文件之一，主线差异非常明显：

- `algo-新版0604` 中存在整套 `scenario_exploration.yaml` 解析与默认配置：
  - `online_threshold_split`
  - `post_run_rolling_drift_analysis`
  - `online_threshold_calibration`
  - `distribution_balance_guard`
- 当前项目移除了上面这些“探索配置文件驱动”的解析与归一化逻辑。
- 当前项目删除了 `single_attack_types` 相关参数与约束传递。
- 当前项目将阈值目标默认值改为更直接的固定值：
  - `threshold_objective` 默认 `f1`
  - `threshold_min_precision` 默认 `0.5`
- 当前项目新增了“accuracy guard（精度守卫）”能力：
  - 新增 `--enable-accuracy-guard`
  - 新增 `--min-failure-detection-accuracy`
  - 在最终输出前，如果检测精度低于阈值，则只针对最终阈值做轻量重调
  - 将守卫前后的阈值、精度、混淆矩阵写入 `output_summary.txt`
- 当前项目删除了大量与 `rolling_drift_analysis`、`distribution_balance_guard`、`threshold_support_guard` 相关的输出字段。
- 覆盖率提前停止逻辑的注释也被简化，当前代码保留的是更直接的控制分支。

结论：

- `algo-新版0604` 更像“参数化更强、可调项更多”的版本。
- 当前项目更像“收敛为固定策略 + 增加精度兜底”的版本。
- 这部分不建议机械覆盖，需要按业务目标做选择性合并。

### 4.1.3 `iterative_testing/run_batch_experiments.py`

当前项目删除了“单攻击白名单”的一整套能力：

- 删除 `--single-attack-types` 参数。
- 删除 `ATTACK_NAME_TO_FIELD`、`ATTACK_FIELD_SET`。
- 删除 `normalize_single_attack_types(...)`。
- `is_attack_combination_valid(...)` 从：
  - 最多一个攻击字段启用
  - 可按白名单限制允许的攻击类型
  
  变为：
  - 仅约束“最多一个攻击字段 > 0”
- `run_batch(...)` 不再接收 `single_attack_types`。

结论：

- 如果后续需要保留“单攻击类型白名单探索”，该文件必须从 `algo-新版0604` 回收相关逻辑。

### 4.1.4 `failure_and_attribution_analysis/scenario_parameter_generator.py`

当前项目删除了与单攻击白名单直接耦合的逻辑：

- 删除对 `ATTACK_FIELD_SET`、`normalize_single_attack_types` 的导入。
- 删除构造器参数 `single_attack_types`。
- 删除生成阶段对“允许攻击字段集合”的筛选。
- 当前项目仅保留“同一场景最多保留一个最强攻击”的裁剪逻辑。

结论：

- 当前项目的场景生成器已经不再支持“只允许某几类攻击被探索”的能力。
- 这与 `run_batch_experiments.py`、`iterative_failure_simulation.py` 的收缩方向一致。

## 4.2 中优先级：运行策略与阈值行为调整

### 4.2.1 `online_self_healing/self_healing_module/healing_orchestrator.py`

有两处行为差异：

- `FAILSCORE_LOW_LOWER`
  - `algo-新版0604`: `0.0001`
  - 当前项目: 一个极小负数 `-0.00000000000000000000001`
- healthy 样本抽样上限
  - `algo-新版0604`: `80`
  - 当前项目: `40`

可能影响：

- 低 fail score 分层边界会更宽松。
- 健康样本抽样数量减半，可能影响自愈阶段的参考样本覆盖度与速度。

### 4.2.2 `env_config.md`

默认环境参数发生了实质变化，当前项目与 `algo-新版0604` 不同：

- `ConstellationConfig`: `0 -> 2`
- `DegradedEdgeRatio`: `0.237 -> 0.08`
- `EdgeDisconnectRatio`: `0.003 -> 0.1`
- `EdgeBandwidthMeanDecreaseRatio`: `0.003 -> 0.2`
- `EdgeBandwidthDecreaseStd`: `0.016 -> 0.1`
- 当前项目新增 `TrafficProfile: "low"`
- 当前项目删除了：
  - `PoissonRate`
  - `MeanIntervalTime`
  - `PacketGenerationInterval`
- `PacketSizeMean`、`PacketSizeStd` 也改成了新的默认值

结论：

- 当前项目默认实验环境已经明显偏向另一套场景设定。
- 如果后续要做“代码合并但保持新版0604运行语义”，这个文件不能直接沿用当前版本。

## 5. 合并建议

建议按下面的优先级处理：

1. 先确认要保留哪条主线
   - A. 保留 `algo-新版0604` 的“探索配置文件驱动 + 单攻击白名单 + rolling drift / distribution balance”
   - B. 保留当前项目的“简化命令链路 + accuracy guard + 指标回填”
   - C. 做混合版：恢复配置驱动，同时保留 accuracy guard

2. 若目标是“尽量合并两边能力”，建议优先保留这些内容
   - 保留当前项目的 `accuracy guard`
   - 保留当前项目在 `run_full_project_pipeline.py` 中新增的 `latest_coverage` / `failure_detection_accuracy` 指标回填
   - 从 `algo-新版0604` 回收 `scenario_exploration.yaml`
   - 从 `algo-新版0604` 回收 `single_attack_types` 相关链路
   - 评估是否恢复 `rolling_drift_analysis` 与 `distribution_balance_guard`

3. 若目标是“以当前项目为主，仅吸收新版0604缺失能力”，建议优先补回
   - `failure_and_attribution_analysis/config/scenario_exploration.yaml`
   - `failure_and_attribution_analysis/COVERAGE_SEMANTICS.md`
   - 单攻击白名单支持链路

## 6. 建议的后续动作

建议下一步按以下顺序推进：

当前项目不是在 `algo-新版0604` 基础上的简单增量版，而是沿着“减少探索配置入口、弱化单攻击白名单、增加 accuracy guard 与结果指标回填”的方向继续演化了；因此后续应做“选择性合并”，不建议直接整目录覆盖。
