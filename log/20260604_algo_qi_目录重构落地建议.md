# algo-qi 目录重构落地建议

本文档基于 [20260604_refactor_migration_guide.md](/abs/e:/project/code/algo-qi/log/20260604_refactor_migration_guide.md:1) ，用于给 `algo-qi` 当前仓库提供一份更贴近现状的落地方案。

结论先说：

- 原迁移指南的原则是对的，不建议推翻重写
- 但它目前更像“通用方法论”，还不是 `algo-qi` 的直接施工单
- 对 `algo-qi` 来说，建议保留原指南不改，额外补这一份“本仓库专用映射与执行顺序”

## 1. 当前仓库状态判断

当前仓库不是一个“纯旧结构、尚未开始重构”的状态，而是“已经出现部分新结构，但主流程仍大量依赖旧路径”的过渡态。

目前已经存在的重构痕迹：

- 顶层已有 `data/`
- 顶层已有 `src/`
- `data/tmp_data/` 已经存在
- `data/model_weights/` 已经存在

但同时仍保留大量旧结构与旧引用：

- `failure_and_attribution_analysis/` 仍同时承载源码和产物语义
- `iterative_testing/`、`online_self_healing/`、`tests/` 仍在根目录
- `env_config.md` 仍在根目录
- `train/` 仍在根目录
- 多个脚本仍直接拼接 `failure_and_attribution_analysis/full_project_runs`、`.../data_archive`、`.../model_artifacts`
- 多个脚本仍各自手算 `PROJECT_ROOT`

因此，`algo-qi` 当前最适合采用的策略不是“直接照搬目标目录”，而是：

- 先做路径治理
- 再做物理迁移
- 最后收口旧目录

## 2. 对原指南的适用结论

原指南里以下原则可以直接照用，不需要修改：

- 先定职责，再迁目录
- 先治路径，再搬代码
- 静态资源要有唯一正式来源
- 运行产物要统一出口
- 子进程路径要单独排查
- 历史数据不强行改写，只要求新数据落到新位置

但以下几点需要按 `algo-qi` 现状补充解释：

### 2.1 `train/` 不能直接按旧指南粗暴归档到 `data`

当前仓库中的 `train/*.yaml` 不是纯历史归档，更像“仍在被主流程直接使用的活动配置”。

例如：

- [run_full_project_pipeline.py](/abs/e:/project/code/algo-qi/run_full_project_pipeline.py:40) 默认读取 `train/train_NewDDQN_dueling_shuffle.yaml`
- [iterative_testing/iterative_failure_simulation.py](/abs/e:/project/code/algo-qi/iterative_testing/iterative_failure_simulation.py:191) 也默认读取同类训练配置

因此对 `algo-qi` 更合理的做法是：

- 当前活跃训练配置迁入 `config/training/`
- 真正历史归档或样板集合再放入 `data/train_config_archive/`

### 2.2 根目录入口不建议一开始迁入 `src`

原指南本身也强调，根目录常用入口可以保留真实实现。

对 `algo-qi` 来说，现阶段建议保留：

- `run_full_project_pipeline.py`

原因：

- 它是主入口
- 它串联了多个子模块和子进程
- 在路径治理完成前，不适合同时再叠加一次入口迁移风险

### 2.3 `src/` 已存在，但暂时还是空壳

这说明仓库已经开始尝试向新结构靠拢，但还没有真正完成代码迁移。

因此不要因为 `src/` 已存在，就默认“源码只差搬过去”。在真正搬源码前，必须先把导入方式、子进程入口、资源路径统一掉。

## 3. algo-qi 建议目标结构

建议把 `algo-qi` 的目标结构收敛为：

- `src/`
- `data/`
- `config/`
- `log/`
- 少量根目录入口脚本

其中建议进一步细化为：

- `src/iterative_testing`
- `src/failure_and_attribution_analysis`
- `src/online_self_healing`
- `src/tests`
- `src/project_paths.py`
- `config/environment/env_config.md`
- `config/training/*.yaml`
- `data/model_weights`
- `data/TLE`
- `data/visual_data`
- `data/tmp_data/full_project_runs`
- `data/tmp_data/smoke_runs`
- `data/tmp_data/closed_loop_outputs`
- `data/tmp_data/data_archive`
- `data/tmp_data/model_artifacts`

说明：

- `data/TLE`、`data/visual_data` 是当前仓库已有资源，应继续视为静态资源目录
- `failure_and_attribution_analysis/2.2module/attack_classifier` 与 `attribution_analysis` 内如果是模型或产物，要继续区分“代码”和“模型文件”

## 4. 当前目录到目标目录的建议映射

建议映射如下：

- `iterative_testing/` -> `src/iterative_testing/`
- `failure_and_attribution_analysis/` -> `src/failure_and_attribution_analysis/`
- `online_self_healing/` -> `src/online_self_healing/`
- `tests/` -> `src/tests/`
- `env_config.md` -> `config/environment/env_config.md`
- `train/*.yaml` -> `config/training/*.yaml`

运行产物统一收口为：

- `failure_and_attribution_analysis/full_project_runs/` -> `data/tmp_data/full_project_runs/`
- `failure_and_attribution_analysis/smoke_runs/` -> `data/tmp_data/smoke_runs/`
- `failure_and_attribution_analysis/closed_loop_outputs/` -> `data/tmp_data/closed_loop_outputs/`
- `failure_and_attribution_analysis/data_archive/` -> `data/tmp_data/data_archive/`
- `failure_and_attribution_analysis/2.2module/model_artifacts/` -> `data/tmp_data/model_artifacts/`
- `training_process_data/` -> `data/tmp_data/training_process_data/`

需要特别判定唯一来源的资源：

- `model_weights` 正式来源建议统一到 `data/model_weights/`
- `Ground_Data`、`Satellite_Data` 若当前项目仍有旧引用，应统一切到 `data/` 下正式目录
- `online_self_healing/model_weight` 若存在或后续恢复出现，应视为待淘汰副本

## 5. 当前仓库已暴露出的重构阻塞点

以下问题意味着现在还不能直接进入“大搬家”阶段。

### 5.1 主流程仍硬编码旧产物目录

[run_full_project_pipeline.py](/abs/e:/project/code/algo-qi/run_full_project_pipeline.py:74) 仍把默认运行目录指向：

- `failure_and_attribution_analysis/full_project_runs`

同文件还直接引用：

- `failure_and_attribution_analysis/2.2module/run_2_2module_pipeline.py`
- `failure_and_attribution_analysis/config/scenario_exploration.yaml`
- `failure_and_attribution_analysis/data_archive`

这说明它还处于“旧结构驱动”状态。

### 5.2 子流程也在使用旧目录

[iterative_testing/iterative_failure_simulation.py](/abs/e:/project/code/algo-qi/iterative_testing/iterative_failure_simulation.py:200) 默认输出到：

- `failure_and_attribution_analysis/closed_loop_outputs`

同文件还默认读取：

- 根目录 `env_config.md`
- `failure_and_attribution_analysis/config/scenario_exploration.yaml`
- 根目录 `train/...yaml`

### 5.3 2.2 模块大量写死旧路径

当前 2.2 模块配置和脚本仍直接写：

- `failure_and_attribution_analysis/full_project_runs`
- `failure_and_attribution_analysis/data_archive`
- `failure_and_attribution_analysis/2.2module/model_artifacts/...`

这类路径既出现在脚本里，也出现在 `pipeline_config.jsonc` / `pipeline_config.template.jsonc` 里。

### 5.4 多个模块仍在各自手算 `PROJECT_ROOT`

当前至少这些文件仍在自己推导根目录：

- [run_full_project_pipeline.py](/abs/e:/project/code/algo-qi/run_full_project_pipeline.py:26)
- [iterative_testing/iterative_failure_simulation.py](/abs/e:/project/code/algo-qi/iterative_testing/iterative_failure_simulation.py:19)
- [iterative_testing/run_batch_experiments.py](/abs/e:/project/code/algo-qi/iterative_testing/run_batch_experiments.py:15)
- [online_self_healing/pipeline_integration/runner.py](/abs/e:/project/code/algo-qi/online_self_healing/pipeline_integration/runner.py:26)

在目录尚未稳定时，这会导致未来每移动一次目录，就要重新修一轮根路径推导。

### 5.5 静态资源来源仍不统一

例如：

- [iterative_testing/PRC.py](/abs/e:/project/code/algo-qi/iterative_testing/PRC.py:87) 仍默认读取根目录 `Satellite_Data`
- [online_self_healing/framework.py](/abs/e:/project/code/algo-qi/online_self_healing/framework.py:443) 仍从模块目录找 `Satellite_Data`
- [online_self_healing/train_NewDDQN_dueling_shuffle.yaml](/abs/e:/project/code/algo-qi/online_self_healing/train_NewDDQN_dueling_shuffle.yaml:139) 仍引用 `./Ground_Data/...`

这说明“唯一正式静态资源来源”目前还没有落稳。

## 6. 推荐执行顺序

对 `algo-qi` 建议按下面顺序推进。

### 阶段 1：只做路径治理，不搬源码

1. 新建 `src/project_paths.py`
2. 在里面统一定义：
   - `PROJECT_ROOT`
   - `SRC_ROOT`
   - `DATA_ROOT`
   - `TMP_DATA_ROOT`
   - `CONFIG_ROOT`
   - `LOG_ROOT`
   - 常用静态资源目录
   - 常用运行产物目录
3. 先把主流程、子流程、2.2 模块中的默认路径全部改成从 `project_paths.py` 派生
4. 先让“新生成结果”真正写入 `data/tmp_data/...`

### 阶段 2：统一静态资源读取口

1. 明确 `Ground_Data`、`Satellite_Data`、模型权重的唯一正式来源
2. 修正 `PRC.py`、`framework.py`、yaml 配置中的旧资源路径
3. 确认不再从模块内副本读取资源

### 阶段 3：迁移配置

1. 新建 `config/environment/`
2. 迁移 `env_config.md` 到 `config/environment/env_config.md`
3. 新建 `config/training/`
4. 迁移当前仍在使用的 `train/*.yaml`
5. 修正所有默认配置路径

### 阶段 4：迁移源码到 `src`

1. 迁移 `iterative_testing/`
2. 迁移 `failure_and_attribution_analysis/`
3. 迁移 `online_self_healing/`
4. 迁移 `tests/`
5. 统一修正导入与子进程脚本路径

### 阶段 5：收口旧目录

1. 清理已不再使用的旧产物目录
2. 清理重复静态资源副本
3. 评估是否保留最少量兼容壳

## 7. 对本项目最重要的验证点

完成每一阶段后，至少验证以下事项：

- `run_full_project_pipeline.py` 能否继续作为主入口运行
- `iterative_failure_simulation.py` 子进程是否仍能拉起
- 2.2 模块是否仍能读取分析输入
- 新生成的结果是否进入 `data/tmp_data/...`
- 报告内部记录的路径是否也已经变成新路径
- `accuracy guard` 相关链路是否不受目录迁移影响

最后这一点要单独关注，因为当前项目相比参考版本新增了这部分行为，不能在目录重构时把它顺手“重构掉”。

## 8. 最终建议

对当前这份通用迁移指南，我的建议是：

- 不用改原文档主体
- 直接保留它作为“总原则文档”
- 在 `algo-qi` 中补充本仓库专用落地文档

换句话说：

- 原指南可以继续执行
- 但执行前，必须先按本文件补齐 `algo-qi` 自己的映射和阶段顺序
- 不建议拿原指南就直接开始整体搬目录

如果后续正式开始施工，第一步最值得做的不是移动目录，而是先落一个统一路径模块 `src/project_paths.py`，然后把主流程和子进程的默认路径收敛进去。
