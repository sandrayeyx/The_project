# 0608 压力分级与低压分支实施设计

## 问题描述

当前系统默认把所有场景都放进同一套失效识别链路中处理：

- 高压场景下，攻击影响更强，指标退化明显，`single_fused_score` 更容易形成有效判别。
- 低压场景下，环境本身较稳定，真实失效样本少，失效边界更弱，`single_fused_score` 容易因为正样本稀缺而冻结。

离线重算已经证明：

1. 低失效率 fallback 已经能把最终判定从 fused 自动退回双阈值；
2. 但在低压场景中，双阈值本身也可能抓不住失效；
3. 这说明问题已经不只是“fused 样本不够”，而是**高压和低压本质上不是同一个判别问题**。

因此需要把“压力等级识别”和“分压力等级失效识别”拆开，建立两级模型策略。

## 背景 / 输入信息

- 当前高压链路已经存在：
  - `failure_score_v2`
  - `decision_score_v2`
  - `single_fused_score`
  - `low_failure_regime fallback`
- 当前主要问题集中在低压环境：
  - 失效样本天然少
  - fused support 难以积累
  - 双阈值在低压环境中的排序能力也未必可靠
- 用户希望优先考虑：
  - 一级：压力等级识别器
  - 二级：高压走 fused，低压走单独路线

## 采用方案

采用“两级判定架构”：

### 一级：压力等级识别器

目标：
- 在进行失效判定之前，先把样本划入 `high_pressure` 或 `low_pressure`

### 二级：按压力分路

- `high_pressure`
  - 继续沿用现有 `single_fused_score`
  - 保留现有 low-failure fallback 作为兜底

- `low_pressure`
  - 不直接走 fused 主链路
  - 先建立一个**低压专用分类器**
  - 第一版优先采用**代价敏感分类 Cost-sensitive Learning**

## 为什么这样选

### 不直接先上 Focal Loss

Focal Loss 更适合作为低压分类器的增强项，而不是第一步。

原因：
- 你现在最需要先解决的是“低压场景要不要单独建模”
- 代价敏感分类实现更简单，验证因果更清晰
- Focal Loss 可以在第二阶段作为 low-pressure classifier 的训练增强

### 不直接先上异常检测

异常检测适合作为低压分支的辅助信号，不适合作为第一版主判定。

原因：
- 异常不一定等于失效
- 低压环境中可能出现“轻微异常但未失效”
- 如果直接把异常分数等同于失效，误报会偏高

因此建议：
- 第一版：低压主分类器
- 第二版：异常检测作为辅助分数或灰区仲裁器

## 可实施设计

### 一、一级压力等级识别器

#### 目标定义

压力等级识别器不是判断“有没有失效”，而是判断：

- 当前样本属于“高压判别问题”
- 还是“低压判别问题”

这是一个**场景分型器**，不是失效分类器。

#### 建议输入特征

优先使用**场景参数 + 攻击配置 + 稳态统计特征**，不要一开始就喂全量 step 时序。

第一版建议输入分三类：

1. 场景基础压力特征
   - `ConstellationConfig`
   - `TrafficProfile`
   - `PoissonRate`
   - `MeanIntervalTime`
   - `PacketGenerationInterval`
   - `PacketSizeMean`
   - `PacketSizeStd`

2. 环境扰动强度特征
   - `DegradedEdgeRatio`
   - `EdgeDisconnectRatio`
   - `EdgeBandwidthMeanDecreaseRatio`
   - `EdgeBandwidthDecreaseStd`

3. 攻击强度特征
   - 各攻击 level
   - 或攻击强度聚合特征，例如：
     - `attack_type_count`
     - `attack_level_sum`
     - `attack_level_max`

可选第四类：
4. 运行后的稳态统计特征
   - `converged_mean_v2`
   - `terminal_risk_score`
   - `decision_score_v2`

第一版建议**不要混太多后验输出特征**，否则压力分级器容易退化成另一个失效分类器。

#### 标签定义

第一版不要一上来用监督学习硬标“高压/低压”。

建议采用**规则标签**先做：

- `high_pressure`：
  - 环境扰动比例较高
  - 或攻击强度较高
  - 或历史统计上该类场景失效率明显高

- `low_pressure`：
  - 环境扰动比例低
  - 攻击强度低到中等
  - 且健康样本占比高、真实失效率低

更实际的做法：
- 第一版先用规则直接划分压力等级
- 暂时**不训练压力等级识别模型**
- 等后续样本累计够，再把规则标签拿去训练一级分类器

也就是说：

### 第一版实现优先级

不是“先训练一个压力识别器”，而是：

**先把压力等级识别写成规则路由器**

这样风险更低，也更容易解释。

#### 第一版压力规则建议

建议先做一个简单可解释的压力评分：

`pressure_score = 环境扰动强度 + 攻击强度`

例如：

- 环境强度部分：
  - `DegradedEdgeRatio`
  - `EdgeDisconnectRatio`
  - `EdgeBandwidthMeanDecreaseRatio`

- 攻击强度部分：
  - 各 attack level 的归一化和

然后根据规则：

- `pressure_score >= high_threshold` -> `high_pressure`
- 否则 -> `low_pressure`

第一版不要分三档，先只分：
- `low_pressure`
- `high_pressure`

这样实现最简单。

思考：现在的一个问题是可能最开始的数据量不够，无法支持训练压力识别器，我觉得规则路由器可以。但我目前有个想法是是否可以针对第一轮场景生成的判别作为辅助，对于失效样本的比例纳入考虑中，如果比例大，那么大概率是高压，如果比例小，则大概率是低压。但是可能也存在即使是低压环境但是正好环境触发关键节点，导致失效样本高。但这种情况感觉也是走高压场景的后续流程比较好，我该如何抉择？

### 二、二级高压分支

#### 目标

高压分支继续使用当前成熟链路，不重写。

#### 行为

- 主模式：`single_fused_score`
- 样本不足时：保留现有 `low_failure_regime` fallback
- 不额外引入新模型

#### 为什么高压保留 fused

因为高压场景下：
- 指标变化更明显
- 失效样本更多
- fused 的训练支持通常更容易积累

所以没必要在高压场景里推翻当前路线。

### 三、二级低压分支

#### 第一版目标

先构建一个**低压专用分类器**，用和高压不同的训练策略。

#### 第一版模型建议

优先选：

- 线性模型 / MLP 小模型 / 树模型 三选一

从最小改动角度看，优先级建议：

1. 复用现有 low-dimensional 表征，做一个**低压 MLP**
2. 如果想先简单稳一点，可以先上**逻辑回归 / 线性分类器**
3. 树模型可作为备选，但会引入新的训练依赖和风格差异

如果目标是尽量少改现有代码，建议：

**第一版低压分支继续用现有 PyTorch 训练框架，单独加一个 low-pressure classifier**

#### 第一版训练策略

第一版低压分类器优先用：

### 代价敏感分类 Cost-sensitive Learning

做法：
- 健康类权重：`1`
- 失效类权重：按样本比例或试验值提高

建议先测：
- `1:10`
- `1:20`
- `1:50`

如果使用 BCE：
- 使用 `pos_weight`

如果使用 sklearn / 线性模型：
- 使用 `class_weight`

#### 为什么第一版先用代价敏感分类

- 不改变原始数据分布
- 对少数失效样本更友好
- 最符合你当前“低压场景失效样本少，但不是完全没有”的事实
- 非常适合做 first baseline

#### 低压分支阈值

低压分支不要默认沿用高压阈值。

建议：
- 低压分支独立校准阈值
- 并允许阈值比高压分支更宽松

例如：
- 高压阈值：偏保守
- 低压阈值：偏提高召回

目标上建议：
- 低压场景优先看 `recall_at_precision`
  - 例如保证 precision 不低于一个底线，再尽量提高 recall

而不是继续只看 `balanced_accuracy`。

### 四、异常检测的放置位置

不建议第一版就把异常检测作为主分类器。

建议放在第二版作为辅助层：

#### 方案 A：灰区仲裁器

当 low-pressure classifier 的输出概率落在灰区时：
- 例如 `0.4 ~ 0.6`
- 再参考 anomaly score

#### 方案 B：预警器

异常检测不直接给最终失效标签，只输出：
- `anomaly_risk_score`
- `low_pressure_warning_flag`

这样既能利用健康样本多的优势，又不会把异常直接等同于失效。

### 五、代码接入建议

尽量少改主流程时，建议新增一层“分路决策”，不要把现有 fused 逻辑拆散。

#### 改动点 1：压力等级判定

在 sample-level summary record 形成后，增加：

- `pressure_regime`
- `pressure_score`

第一版先由规则函数给出，例如：

- `_compute_pressure_regime(record) -> (regime, score)`

#### 改动点 2：训练时按压力分路

在失效判定训练前：
- 如果 `pressure_regime == high_pressure`
  - 走现有 fused
- 如果 `pressure_regime == low_pressure`
  - 走 low-pressure classifier

#### 改动点 3：最终推理时按压力分路

在 `_recompute_predictions_from_thresholds()` 或其上一层：
- 先看 `pressure_regime`
- 再选择最终使用的模型/阈值

建议增加字段：
- `effective_pressure_regime`
- `effective_decision_mode`

示例：
- `high_pressure + single_fused_score`
- `high_pressure + dual_threshold_v2`
- `low_pressure + low_pressure_classifier`

#### 改动点 4：日志输出

输出中新增：
- `pressure_regime`
- `pressure_score`
- `low_pressure_classifier_status`
- `low_pressure_classifier_holdout_auc`
- `effective_decision_mode`

这样后续就能直接分析：
- 哪些样本被分到低压
- 低压分支效果如何
- 是否真的比当前 fallback 更有效

## 推荐实施顺序

### V1：最小可验证版

1. 新增压力规则路由器
2. 高压保留 fused
3. 低压新增代价敏感分类器
4. 低压分支独立阈值
5. 完整打日志

这是最推荐先做的一版。

### V2：增强版

在低压分支中加入：
- Focal Loss

适用条件：
- 第一版低压分类器仍明显被健康样本压制

### V3：辅助版

加入异常检测作为：
- 灰区仲裁器
- 或预警器

不作为第一主分类器。

## 为什么这是当前最稳的路线

1. 先承认高压和低压不是同一种判别问题
2. 不推翻现有 fused 高压主链路
3. 低压分支优先选择实现简单、解释性强的代价敏感分类
4. 把 Focal Loss 和异常检测都放在第二阶段，不让第一版复杂度失控

## 进一步细化（便于逐点批注）

### 1. 一个关键约束：攻击强度标签不能直接等价于压力等级

需要明确记录当前数据生成方式的一个现实约束：

- 除了被额外补入的健康样本
- 以及最开始的训练/初始基线场景
- 其余大多数攻击样本的 attack level 基本是**从 1 到 4 随机采样**

这意味着：

1. `Attack_level` 本身不完全等价于“压力强度”
   - 因为同样是 level 4，所处环境压力可能完全不同
   - 同样是 level 1，在低压环境里也可能很难判别

2. 如果一级压力分级器过度依赖 `attack_level_sum` 或 `attack_level_max`
   - 很容易退化成“按攻击数字大小分组”
   - 而不是按“场景本身是不是高压判别问题”分组

3. 因此第一版压力分级中：
   - `attack_level` 只能作为**辅助特征**
   - 不能作为主导特征

这条约束非常重要，后续如果你有自己的看法，建议优先围绕这一点批注。

### 2. 一级压力分级器第一版建议：规则路由器，不先训练模型

基于上面的约束，第一版不建议直接训练“高压/低压分类器”，而是先做一个**规则路由器**。

原因：

- 攻击 level 的随机性会污染一级监督标签
- 当前阶段更需要可解释性，而不是复杂度
- 规则路由器更方便你后续逐项改权重、改阈值

第一版目标不是得到一个最优压力分类器，而是先把这两个问题拆开：

- 哪些样本应该继续走 fused
- 哪些样本应该送去低压专用分支

### 3. `pressure_score` 的第一版设计建议

#### 3.1 设计原则

第一版 `pressure_score` 建议遵循三条原则：

1. **环境扰动强度优先**
   - 比 attack level 更可信

2. **攻击特征只做弱修正**
   - 因为它们很多时候是随机采样出来的

3. **允许少量后验统计，但不能完全依赖失效分数本身**
   - 否则一级分流器会变成伪装的失效分类器

#### 3.2 第一版特征分组

建议把 `pressure_score` 分成三部分：

##### A. 环境压力主分量

这是主分量，建议权重最高。

候选特征：
- `DegradedEdgeRatio`
- `EdgeDisconnectRatio`
- `EdgeBandwidthMeanDecreaseRatio`
- `EdgeBandwidthDecreaseStd`
- `PoissonRate`
- `PacketGenerationInterval` 的反向强度

解释：
- 这些特征更接近“环境本身有多恶劣”
- 比 attack level 更适合做一级分流

##### B. 攻击扰动辅分量

这是辅分量，建议权重明显低于环境分量。

候选特征：
- `attack_type_count`
- `attack_level_sum_norm`
- `attack_level_max_norm`

但需要特别注意：

- 这里只能作为“扰动存在程度”的弱信号
- 不能让它盖过环境压力主分量

建议第一版做法：

- 只取 `attack_type_count`
- 或 `attack_level_max / 4`
- 且整体权重较低

#### 3.3 第一版可执行公式（建议草案）

建议先做一个简单的线性规则，方便你后面手改：

```text
pressure_score =
  0.35 * DegradedEdgeRatio
  + 0.30 * EdgeDisconnectRatio
  + 0.20 * EdgeBandwidthMeanDecreaseRatio
  + 0.10 * EdgeBandwidthDecreaseStd_norm
  + 0.05 * attack_signal
```

其中：

- `EdgeBandwidthDecreaseStd_norm`
  - 先按历史范围或配置范围归一化到 `[0, 1]`
- `attack_signal`
  - 第一版建议用 `attack_level_max / 4`
  - 或者更保守一点，直接用 `has_attack ? 1 : 0`

因为你的 attack level 大多是随机 1-4，我更推荐：

### 第一版 `attack_signal`

```text
attack_signal = 1  (有攻击)
attack_signal = 0  (无攻击)
```

这样它只表示“有没有攻击”，不直接把随机等级当成压力大小。

### 4. 高压 / 低压阈值第一版怎么定

#### 4.1 不建议一开始分三档

第一版先只分：

- `low_pressure`
- `high_pressure`

不要一开始就加 `mid_pressure`，否则会把分路逻辑复杂化。

#### 4.2 阈值来源建议

第一版阈值不要凭感觉写死，建议从现有旧数据里看分布后定一个经验阈值。

建议做法：

1. 收集一批已有样本
2. 计算每条样本的 `pressure_score`
3. 观察：
   - fused 明显有效的那批样本，`pressure_score` 大概落在哪
   - fused 经常冻结、双阈值也吃力的样本，`pressure_score` 落在哪
4. 用这个分布先切一个经验阈值

第一版可暂定：

```text
pressure_score >= 0.45 -> high_pressure
pressure_score < 0.45 -> low_pressure
```

但这个 `0.45` 只是建议起点，不是最终答案。

更重要的是：

- 先把 `pressure_score` 和 `pressure_regime` 打进日志
- 让你能观察它到底把哪些样本送去了低压分支

#### 4.3 第一版批次修正规则（已确认可行）

结合当前讨论，第一版建议采用：

### 静态初判决定当前批次

即：
- 当前批次先根据 `pressure_score` 得到 `initial_pressure_regime`
- 当前批次直接按这个 regime 进入对应分支
  - `high_pressure -> fused`
  - `low_pressure -> low-pressure classifier`

### 批次失效比例修正只影响下一轮

即：
- 当前批次跑完后，再统计：
  - `batch_failure_ratio`
  - `batch_high_risk_ratio`
  - 可选：`terminal_hard_failure_ratio`
- 如果这些比例明显偏高，则产生：
  - `pressure_override_signal`
- 该信号**不回头重判当前批次**
- 只用于修正：
  - 下一轮同类样本的默认路由
  - 或下一批相似场景的 pressure regime 初始判断

#### 为什么第一版这样做

这样设计的优点是：

1. 当前批次语义清晰
   - 不会出现“一批样本先低压判一遍，又整体切高压重判一遍”的混乱

2. 不会形成太强的自我强化偏差
   - 不直接用“本批模型判出来的结果”再回头改本批模型路径

3. 实现复杂度更低
   - 第一版不需要支持当前批次双重推理和覆盖式重算

#### 对用户担心场景的解释

即使一个名义低压场景，恰好触发关键节点，导致这一批失效比例明显升高：

- 第一版中，这一批仍按静态初判路线跑完
- 但该批次结果会把后续同类样本上调为高压路线

换句话说：

**第一版允许“批次反馈改变后续路由”，但不允许“批次反馈回头推翻当前批次”。**

这条是当前推荐默认规则。

#### 4.3 第一版建议定稿（当前采用）

结合前面的讨论，第一版先采用**纯环境压力评分**，不把 `attack_level` 纳入主公式。

这样做的原因是：

- 当前攻击强度很多时候是随机采样 1-4，稳定性不够
- 第一版更需要可解释、可观察的分流依据
- 压力分流的目标是识别“当前是不是高压判别问题”，而不是直接编码攻击数字大小

第一版建议公式如下：

```text
pressure_score =
  0.40 * DegradedEdgeRatio
  + 0.35 * EdgeDisconnectRatio
  + 0.20 * EdgeBandwidthMeanDecreaseRatio
  + 0.05 * EdgeBandwidthDecreaseStd_norm
```

说明：

- `EdgeBandwidthDecreaseStd_norm` 需要先归一化到 `[0, 1]`
- 第一版 `pressure_score` 只承担静态初判作用
- 第一版不把 `attack_level` 纳入主公式

静态初判规则建议定为：

```text
if pressure_score >= 0.45:
    initial_pressure_regime = "high_pressure"
else:
    initial_pressure_regime = "low_pressure"
```

这里的 `0.45` 作为第一版经验阈值起点，后续可根据日志分布再校正。

#### 4.4 批次修正 V1 建议定稿（当前采用）

第一版 `pressure_override_signal` 只用于影响**下一轮 / 下一批**的路由，不回溯修改当前批次结果。

也就是说：

- `initial_pressure_regime` 决定当前批次走哪条识别链路
- `pressure_override_signal` 只作为后续批次是否上调到高压路线的修正信号

第一版建议定义批次高风险样本为：

```text
high_risk_sample =
  (decision_score_v2 >= 0.30)
  or (terminal_risk_score >= 0.50)
```

批次统计量建议为：

```text
batch_failure_ratio = predicted_failure_count / batch_size
batch_high_risk_ratio = high_risk_sample_count / batch_size
```

第一版批次升级规则建议为：

```text
if batch_failure_ratio >= 0.20:
    pressure_override_signal = "upgrade_to_high"
elif batch_high_risk_ratio >= 0.35:
    pressure_override_signal = "upgrade_to_high"
else:
    pressure_override_signal = "keep"
```

解释：

- 如果当前批次已经有较高失效比例，则说明这个场景簇更像高压判别问题
- 如果当前批次还没有大量最终失效，但高风险样本比例已经偏高，也允许提前上调
- 第一版只更新下一轮 / 下一批路由，不对当前批次进行整批重判

可以对应整理成接近配置的短版表达：

```yaml
pressure_router_v1:
  score_formula: "0.40*DegradedEdgeRatio + 0.35*EdgeDisconnectRatio + 0.20*EdgeBandwidthMeanDecreaseRatio + 0.05*EdgeBandwidthDecreaseStd_norm"
  high_pressure_threshold: 0.45
  note: "v1 uses environment-only pressure score; attack level is excluded from the main score"

pressure_override_v1:
  high_risk_rule: "decision_score_v2 >= 0.30 or terminal_risk_score >= 0.50"
  upgrade_if_batch_failure_ratio_ge: 0.20
  upgrade_if_batch_high_risk_ratio_ge: 0.35
  apply_scope: "next_round_only"
  note: "override does not rewrite current-batch predictions"
```

### 5. 低压分支第一版：优先选低压 MLP，而不是线性模型

虽然线性模型更简单，但结合你现有代码结构，我更推荐：

**第一版低压分支优先做一个 low-pressure MLP**

原因：

1. 你现有训练框架已经是 PyTorch 体系
   - 接一个小 MLP 成本更低

2. 低压场景里边界往往更模糊
   - 线性模型可能过于刚性

3. 可以和后续 Focal Loss 平滑衔接

所以第一版模型优先级建议改成：

1. `low-pressure MLP`
2. 线性分类器（做对照 baseline）
3. 树模型（暂不优先）

### 6. 低压分支训练特征第一版建议

低压分支不要直接吃全部原始场景字段，建议优先吃已经比较稳定的 summary-level 特征。

第一版建议特征：

- `converged_mean_v2`
- `converged_p75_v2`
- `converged_max_v2`
- `converged_std_v2`
- `converged_slope_v2`
- `converged_high_ratio_v2`
- `terminal_risk_score`
- `terminal_score_gap_v2`
- `decision_score_v2`

可选附加：
- `pressure_score`
- 少量环境参数（如 `DegradedEdgeRatio`, `EdgeDisconnectRatio`）

第一版不建议把所有 attack level 原样全塞进去。

原因还是一样：
- attack level 采样太随机
- 会让低压分类器学到“数据生成器习惯”，而不是真实失效边界

### 7. 低压分支的损失函数和类权重建议

第一版建议：

- 先用普通 BCE
- 加 `pos_weight`

建议实验组：

- `pos_weight = 10`
- `pos_weight = 20`
- `pos_weight = 50`

记录指标：

- recall
- precision
- balanced_accuracy
- PR-AUC（如果后面补）

如果你后面发现：
- 低压 MLP 还是明显被健康样本主导

再升级到：
- `Focal Loss + pos_weight`

也就是说：

### 第一版不要同时上 BCE 和 Focal 两套训练

而是：

1. 先 `BCE + pos_weight`
2. 如果效果不够，再换 `Focal Loss`

### 8. 低压分支阈值目标建议

低压分支不建议继续把 `balanced_accuracy` 当唯一目标。

因为你现在更关心的是：
- 不要把稀少的失效全漏掉

所以第一版建议：

- 低压阈值目标优先使用
  - `recall_at_precision`

思考：我觉得可以

例如：
- precision 底线先设 `0.50` 或 `0.60`
- 在这个约束下尽量提高 recall

如果第一版嫌目标函数改动太大，也可以先做折中：

- 仍记录 `balanced_accuracy`
- 但低压分支阈值选择时额外看 `recall`

## 可选增强（在第一版稳定后再考虑）

### 增强 A：当前批次灰区升级，而不是整批重判

如果后面发现“只影响下一轮”仍然不够灵敏，可以加一个更温和的增强：

- 当前批次仍先按静态初判路线跑
- 如果低压分支输出后，发现：
  - `batch_failure_ratio` 偏高
  - 或 `batch_high_risk_ratio` 偏高
- 则只对**灰区样本**补跑高压链路

这里的灰区样本例如：
- `low-pressure classifier` 输出概率落在 `0.4 ~ 0.6`

这样做的好处是：
- 不需要整批重判
- 只升级最不确定的那一小部分样本
- 风险和复杂度都低于“整批翻盘”

### 增强 B：压力分级从规则路由器升级到训练式分型器

等后续样本累计更充分后，可以考虑把：

- 规则 `pressure_regime`

升级成：

- 训练得到的 pressure classifier

但建议前提是：
- 已经积累了足够多不同压力水平的样本
- 并且第一版日志已经证明规则分路大体合理

### 增强 C：低压分支从 BCE + pos_weight 升级到 Focal Loss

适用条件：
- 第一版低压 MLP 仍明显偏向健康样本
- recall 提升不够

建议顺序：
1. 先 `BCE + pos_weight`
2. 再尝试 `Focal Loss + pos_weight`

### 增强 D：异常检测作为低压辅助信号

适用位置：
- 灰区仲裁器
- 或预警器

不建议直接替代低压主分类器。

## 当前结论（更新）

结合本轮讨论，当前推荐的**第一版正式方案**是：

1. 一级继续采用**规则路由器**
2. 当前批次由**静态初判**决定分支
3. 批次失效比例修正**只影响下一轮**
4. 高压继续走 fused
5. 低压新增 `low-pressure MLP + BCE/pos_weight + recall-oriented threshold`
