# 0608 低失效率场景专用决策策略

## 问题描述

当前 `single_fused_score` 链路默认假设：
- fused 训练样本能持续累积；
- threshold calibration 阶段能够拿到足够多的有效样本；
- 训练集内正负类都能同时出现。

但在“环境干扰很少、真正失效样本本来就少”的场景里，这个假设不成立。

典型表现：
- 真实失效样本数长期偏低；
- `effective_support` 达不到 `threshold_min_support`；
- 即使少量失效样本存在，也可能全部落在正类，训练集内缺负类；
- 最终 `single_fused_score` 冻结，整条最终识别链路退化为“全判正常”或维持旧阈值。

因此需要一套**低失效率场景专用决策策略**，目标是在尽量少改主流程的前提下，让系统在“样本足够时用 fused、样本不足时稳定 fallback”。

## 背景 / 输入信息

- 当前主模式：`failure_decision_mode = single_fused_score`
- 当前在线 threshold split 默认配置来自 `scenario_exploration.yaml`
  - `mode: stratified_late_holdout`
  - `holdout_ratio: 0.20`
  - `late_window_ratio: 0.25`
  - `holdout_late_fraction: 0.70`
- 当前在线 threshold calibration 默认目标：
  - `objective: balanced_accuracy`
  - `min_precision: 0.70`
- 当前主流程已有可复用保护：
  - `threshold_min_support`
  - `decision_model_min_support`
  - `support_aware_threshold_guard`
  - `enable_accuracy_guard`

离线检查结果显示，在 `data-analysis/2-healthy` 这类低干扰场景中：
- 有效失效样本只有 `14`
- 小于默认 `threshold_min_support = 30`
- 即使入口规则放宽为方案 B，也没有新增可用样本
- 当前核心矛盾已经从“入口太窄”转成“正样本天然稀少”

## 采用方案

采用“三层决策策略”：

1. **正常模式：优先使用 fused**
   - 当 fused 链路样本支持充分时，继续使用 `single_fused_score`

2. **低失效率 fallback：自动退回双阈值**
   - 当 fused 训练/校准支持不足，或者训练集单类时，不再坚持使用 fused 作为最终判定
   - 自动退回 `decision_score_v2 + terminal_risk_score` 的双阈值机制

3. **小样本保护：只对阈值更新做温和放宽**
   - 不要求在低失效率场景里把 fused 强行学起来
   - 只允许对 support guard 和最小支持数做受控放宽，用于离线试验或专项配置

## 策略设计

### 一、主判定分层

建议把最终决策拆成两个层次：

#### 层 1：候选主模式

默认仍然保留：
- `single_fused_score`

#### 层 2：运行时自动降级

在以下任一条件满足时，自动从 `single_fused_score` 降级到 `dual_threshold_v2`：

1. fused 有效样本数不足
   - `effective_support < threshold_min_support`

2. split 后训练样本不足
   - `train_support < threshold_min_support`

3. 训练集单类
   - `train_positive_count == 0`
   - 或 `train_negative_count == 0`

4. fused 模型未成功拟合
   - `fused_model_status != fitted`
   - 或 `fused_model_holdout_record_count == 0`

5. fused 分数无有效分布
   - 所有 `fused_score` 全相同
   - 或绝大多数 `fused_score = 0`

#### fallback 后的最终判定

使用现有老链路，不引入新模型：

- `decision_score_v2 >= decision_threshold`
- 且 `terminal_risk_score >= terminal_threshold_v2`

这条链路已经存在，不需要发明新逻辑，只需要在 fused 不可用时重新启用。

### 二、推荐 fallback 条件

建议把 fallback 分成两级：

#### Fallback Level 1：训练前 fallback

触发条件：
- `effective_support < threshold_min_support`
- 或 `split_plan.status != updated`
- 或 `train_positive_count == 0`
- 或 `train_negative_count == 0`

动作：
- 跳过 fused threshold calibration
- 不把 fused 作为最终主判定
- 直接切换到 `dual_threshold_v2`

适用意义：
- 避免“明知样本不够还硬上 fused”

#### Fallback Level 2：训练后 fallback

触发条件：
- fused 虽然完成训练，但 `holdout_auc` 极低
- 或 fused score 分布坍缩
- 或 accuracy guard 仍无法把最终准确率拉回最低要求

建议判据：
- `primary_score_holdout_auc < 0.55`
- 或 `predicted_failure_count == 0` 且 `true_failure_count > 0`

动作：
- 保留 fused 训练结果做记录
- 但最终推理口径退回 `dual_threshold_v2`

适用意义：
- 防止 fused 学出一个“技术上已训练，实际上不可用”的空模型

### 三、配置项设计

建议尽量复用现有配置风格，只新增少量开关。

#### 1. 总开关

建议新增：

```yaml
low_failure_regime:
  enabled: "on"
```

作用：
- 只在低失效率场景启用这套 fallback 策略

#### 2. fallback 主策略

```yaml
low_failure_regime:
  enabled: "on"
  fallback_policy: "dual_threshold_v2"
```

可选值建议：
- `dual_threshold_v2`
- `decision_score_only`（不推荐作为默认）

默认建议：
- `dual_threshold_v2`

原因：
- 比仅用单一分数更稳；
- 已有现成实现；
- 更符合“稀有失效时依赖规则/终态双证据”的需求。

#### 3. fallback 触发条件

```yaml
low_failure_regime:
  enabled: "on"
  fallback_policy: "dual_threshold_v2"
  trigger:
    min_effective_support: 30
    require_both_classes_in_train: "on"
    min_fused_holdout_auc: 0.55
    enable_zero_prediction_guard: "on"
```

说明：
- `min_effective_support`
  - 默认直接复用 `threshold_min_support`
  - 不建议单独设更高
- `require_both_classes_in_train`
  - 默认 `on`
  - 低失效率场景下非常关键
- `min_fused_holdout_auc`
  - 用来区分“训练成功”与“训练可用”
- `enable_zero_prediction_guard`
  - 如果 fused 最终一个失效都不报，而真实存在正样本，就允许 fallback

#### 4. 小样本实验配置

只建议给离线实验或专项测试使用：

```yaml
low_failure_regime:
  enabled: "on"
  allow_small_sample_fused_experiment: "off"
  small_sample_threshold_min_support: 12
```

默认建议：
- `allow_small_sample_fused_experiment: off`

原因：
- 正式主流程不建议在小样本场景下强行训 fused
- 但离线分析时可以拿来观察“如果要求没那么严，会不会开始有信号”

### 四、与现有主流程的最小改动落点

尽量少改主流程时，建议只改三个位置。

#### 改动点 1：阈值校准结果出来后，判断 fused 是否可用

位置：
- `_calibrate_failure_threshold_v2()`
- 或其调用后、最终写回 `last_threshold_stats` 前

需要新增的只是一个“是否触发 low-failure fallback”的判断函数。

建议新增行为级 helper：

- `_should_fallback_from_fused(threshold_stats, failure_model_info) -> (bool, reason)`

#### 改动点 2：最终预测阶段支持 fallback

位置：
- `_recompute_predictions_from_thresholds()`

当前逻辑是：
- `single_fused_score` 模式下直接用 `fused_score >= fused_threshold`

建议改成：
- 如果没有触发 fallback，保持原样
- 如果触发 fallback，改用：
  - `decision_score_v2 >= decision_threshold`
  - 且 `terminal_risk_score >= terminal_threshold`

这是最关键、也最省改动的一步。

#### 改动点 3：日志里记录 fallback 状态

输出建议新增字段：
- `low_failure_regime_enabled`
- `low_failure_fallback_applied`
- `low_failure_fallback_reason`
- `effective_decision_mode`

其中：
- `failure_decision_mode` 保留原始配置值，例如 `single_fused_score`
- `effective_decision_mode` 记录本轮真正使用的最终判定方式，例如 `dual_threshold_v2`

这样日志解释会清楚很多。

## 推荐默认策略

如果要一个最实用、最保守的默认版本，我建议：

1. 默认配置仍保留：
   - `failure_decision_mode = single_fused_score`

2. 新增：

```yaml
low_failure_regime:
  enabled: "on"
  fallback_policy: "dual_threshold_v2"
  trigger:
    min_effective_support: 30
    require_both_classes_in_train: "on"
    min_fused_holdout_auc: 0.55
    enable_zero_prediction_guard: "on"
  allow_small_sample_fused_experiment: "off"
```

3. 行为上：
   - 样本够时：继续 fused
   - 样本不够时：自动双阈值 fallback
   - 不强迫 fused 在稀有失效场景里在线学习

## 为什么这套策略适合当前问题

1. **它承认低失效率是场景属性，不把它当异常数据问题**
   - 这很重要

2. **它不推翻你现有 fused 链路**
   - 样本够时还能继续吃到 fused 的收益

3. **它不要求在稀有失效场景里硬凑正样本**
   - 更贴近真实业务

4. **它对主流程侵入小**
   - 主要是在最终决策处增加 fallback 判定
   - 不是重写训练框架
