## 3.3 `failure_and_attribution_analysis/scenario_parameter_generator.py`

### 函数 / 构造参数变化

受影响核心类：

- `ScenarioParameterGenerator`

#### 构造器签名变化

`algo-新版0604` 比当前项目多一个构造参数：

- `single_attack_types: Optional[Sequence[str]] = None`

当前项目删除了该参数。

#### 导入变化

`algo-新版0604` 额外依赖：

- `ATTACK_FIELD_SET`
- `normalize_single_attack_types`

当前项目删除了这两个导入。

### 逻辑变化

受影响的是 `ScenarioParameterGenerator` 内部对攻击字段裁剪的逻辑：

- `algo-新版0604`
  - 支持把 `single_attack_types` 归一化成允许攻击字段集合
  - 在“单攻击模式”下可以限制只允许某几种攻击类型
  - 若生成结果中没有激活攻击，也可能按允许集合补一个攻击字段
  - 若当前最强攻击不在允许集合中，会回退到允许集合中的候选字段
- 当前项目
  - 删除了“允许攻击字段集合”整套逻辑
  - 仅保留“若一个场景激活了多个攻击字段，只保留最强的那个，其他清零”

### 结论

- 当前项目已经不支持“单攻击白名单探索”。
- 这里与 `run_batch_experiments.py`、`iterative_failure_simulation.py` 的收缩方向一致。

## 3.4 `iterative_testing/iterative_failure_simulation.py`

这是差异最大的文件之一。

### 一、`parse_args()` 参数变化

#### 当前项目删除的参数

- `--exploration-config`
- `--single-attack-types`
- `--threshold-split-mode`
- `--threshold-split-holdout-ratio`
- `--threshold-split-late-window-ratio`
- `--threshold-split-holdout-late-fraction`
- `--rolling-drift-analysis`
- `--rolling-window-size`
- `--rolling-step-size`
- `--rolling-min-train-support`
- `--rolling-min-holdout-support`

#### 当前项目新增的参数

- `--enable-accuracy-guard`
- `--min-failure-detection-accuracy`

#### 默认值 / 帮助文本变化

- `--coverage-target`
  - 旧版帮助文本强调“conditional coverage”
  - 当前项目简化为普通数值参数
- `--stop-on-coverage-target`
  - `algo-新版0604` 的帮助文本强调“coverage upper bound”
  - 当前项目帮助文本改成“coverage lower bound reaches target”
  - 但实际主循环逻辑仍需结合代码一起看，不能只看 help
- `--threshold-objective`
  - `algo-新版0604`: 默认空字符串，由探索配置文件决定
  - 当前项目: 默认 `f1`
- `--threshold-min-precision`
  - `algo-新版0604`: 默认 `None`
  - 当前项目: 默认 `0.5`

### 二、顶层函数变化

#### `algo-新版0604` 中存在、当前项目删除的函数

- `parse_on_off_arg`
- `load_exploration_config`
- `normalize_switch_text`
- `_normalize_float`
- `_normalize_int`
- `_merge_nested_mapping`
- `_normalize_threshold_objective`
- `resolve_exploration_settings`
- `resolve_single_attack_types_from_sources`

#### 当前项目新增的函数 / 方法

当前项目新增的是 `ClosedLoopFailureSimulation` 内部与 accuracy guard 相关的方法：

- `_accuracy_guard_score_config()`
- `_select_accuracy_guard_threshold(score_key)`
- `_apply_accuracy_guard()`

### 三、类初始化与状态字段变化

受影响核心类：

- `ClosedLoopFailureSimulation`

#### `algo-新版0604` 中存在的状态字段

- `self.exploration_settings`
- `self.single_attack_types`
- `self.rolling_drift_analysis_config`
- `self.distribution_balance_guard_config`
- `self.rolling_drift_analysis_summary`
- `self.last_distribution_balance_guard_info`
- `self.distribution_balance_guard_state`

#### 当前项目替换成的状态字段

- `self.accuracy_guard_summary`

也就是说，当前项目把：

- `exploration config`
- `single attack whitelist`
- `rolling drift analysis`
- `distribution balance guard`

这一组状态，整体收缩成了 `accuracy guard` 这一条更直接的后处理链路。

### 四、类内部方法变化

#### `algo-新版0604` 中存在、当前项目删除的方法

- `_resolve_distribution_balance_guard_profile()`
- `_apply_distribution_balance_guard(...)`
- `_simulate_post_run_rolling_drift_analysis()`

#### 当前项目新增的方法

- `_accuracy_guard_score_config()`
  - 决定针对哪类 score / threshold 键做精度守卫
- `_select_accuracy_guard_threshold(score_key)`
  - 遍历候选阈值，选择能达到目标精度的阈值
- `_apply_accuracy_guard()`
  - 在最终输出前按 `min_failure_detection_accuracy` 进行阈值回调
  - 回写 `summary_records`
  - 回写 `step_records`
  - 更新 `last_threshold_stats`
  - 产出 `accuracy_guard_summary`

### 五、输出内容变化

#### 当前项目新增写入 `output_summary.txt` 的字段

- `failure_detection_accuracy_raw`
- `accuracy_guard_enabled`
- `accuracy_guard_applied`
- `accuracy_guard_status`
- `accuracy_guard_min_accuracy`
- `accuracy_guard_score_name`
- `accuracy_guard_threshold_before`
- `accuracy_guard_threshold_after`
- `accuracy_guard_confusion_matrix_before`
- `accuracy_guard_confusion_matrix_after`

#### 当前项目删除的输出字段

主要删除两大类：

- `distribution_balance_guard_*`
- `rolling_drift_*`

以及一批阈值拆分与 support guard 相关字段，例如：

- `threshold_split_mode`
- `threshold_split_seed`
- `threshold_split_holdout_ratio`
- `threshold_split_late_window_ratio`
- `threshold_split_holdout_late_fraction`
- `threshold_support_guard_enabled`
- `threshold_support_metric`
- `threshold_support_tier`
- `threshold_min_precision_used`
- `threshold_constraint_status`

### 六、流程行为变化

#### `algo-新版0604`

- 依赖 `scenario_exploration.yaml`
- 支持探索配置归一化
- 支持单攻击白名单
- 支持 distribution balance guard
- 支持 post-run rolling drift analysis

#### 当前项目

- 去掉 `scenario_exploration.yaml` 的运行时耦合
- 去掉单攻击白名单
- 去掉 distribution balance guard
- 去掉 rolling drift analysis
- 新增 accuracy guard，并在结束前执行

### 结论

- 这是“配置驱动探索链路”与“简化主链路 + 精度兜底”的核心分叉点。
- 后续如要合并，两边都不能直接覆盖，需要人工决策。

## 3.6 `iterative_testing/run_batch_experiments.py`

### 一、`parse_args()` 参数变化

#### `algo-新版0604` 独有参数

- `--single-attack-types`

当前项目删除了该参数。

### 二、函数变化

#### `algo-新版0604` 中存在、当前项目删除的常量

- `ATTACK_NAME_TO_FIELD`
- `ATTACK_FIELD_SET`

#### `algo-新版0604` 中存在、当前项目删除的函数

- `normalize_single_attack_types(single_attack_types)`

### 三、函数签名变化

#### `is_attack_combination_valid(...)`

`algo-新版0604`：

```python
is_attack_combination_valid(
    combo,
    template_env,
    single_attack_types=None,
    require_attack=False,
)
```

当前项目：

```python
is_attack_combination_valid(combo, template_env)
```

#### `run_batch(...)`

`algo-新版0604`：

```python
run_batch(project_root, env_md_path, yaml_path, single_attack_types=None)
```

当前项目：

```python
run_batch(project_root, env_md_path, yaml_path)
```

### 四、逻辑变化

#### `algo-新版0604`

- 会把 `single_attack_types` 规范化为攻击字段白名单
- `is_attack_combination_valid(...)` 同时检查：
  - 最多一个攻击字段激活
  - 若要求必须有攻击，则必须有一个攻击字段激活
  - 若配置了白名单，则激活攻击必须在白名单内

#### 当前项目

- 只检查：
  - `count_enabled_attacks(merged_env) <= 1`

### 五、用户提示变化

当没有合法组合时：

- `algo-新版0604` 强调“没有满足攻击约束的组合”
- 当前项目明确成“没有满足攻击约束（同时最多一个攻击字段 > 0）的组合”

### 结论

- 当前项目把批量实验脚本收缩成“只保留单攻击数量约束，不保留攻击类型白名单约束”。

## 3.7 `online_self_healing/self_healing_module/healing_orchestrator.py`

### 参数 / 常量变化

#### 常量变化

- `FAILSCORE_LOW_LOWER`
  - `algo-新版0604`: `0.0001`
  - 当前项目: `-0.00000000000000000000001`

#### 抽样上限变化

在 `TieredHealingOrchestrator` 内部健康样本抽样逻辑中：

- `sample_count = min(sample_count, 80)` 变成
- `sample_count = min(sample_count, 40)`

### 影响

- 低 fail score 分层边界更宽。
- 健康样本最多只取 `40` 个而不是 `80` 个。
- 这会影响自愈阶段候选集规模和分层行为。

## 3.9 `run_full_project_pipeline.py`

这是另一处高优先级差异文件。

### 一、常量 / 配置入口变化

#### `algo-新版0604` 中存在、当前项目删除的常量

- `DEFAULT_EXPLORATION_CONFIG`

对应路径：

- `failure_and_attribution_analysis/config/scenario_exploration.yaml`

### 二、顶层函数变化

#### `algo-新版0604` 中存在、当前项目删除的函数

- `load_exploration_config(path)`
- `resolve_single_attack_types(explicit_list_value, exploration_config_path)`
- `resolve_exploration_cli_override(text)`

#### 当前项目新增的函数

- `read_output_summary_run_metrics(output_summary_path)`

这个函数会从 `output_summary.txt` 中额外提取：

- `latest_coverage_upper_bound`
- `failure_detection_accuracy`

并回填到后续评估记录。

### 三、函数行为变化

#### `resolve_env_config_dict(...)`

`algo-新版0604`：

- 若显式传了 `env_config_json` / `env_config_file`，则走字典配置
- 否则返回 `None`

当前项目：

- 若显式传了 `env_config_json` / `env_config_file`，则走字典配置
- 若设置了 `args.env_md`，则仍返回 `None`
- 否则回退到 `DEFAULT_ENV_CONFIG`

含义：

- 当前项目更偏向保证“总能拿到一份字典环境配置”

#### `build_step_evaluate_rows(...)`

`algo-新版0604`：

- 只写基础场景评估字段

当前项目：

- 新增调用 `read_output_summary_run_metrics(...)`
- 为每条评估记录新增字段：
  - `latest_coverage`
  - `failure_detection_accuracy`

#### `build_simulation_cmd(...)`

`algo-新版0604`：

- 参数签名中包含 `single_attack_types`
- 生成的命令可追加：
  - `--exploration-config`
  - `--single-attack-types`
  - `--threshold-objective`
  - `--threshold-min-precision`
  - `--threshold-split-mode`
  - `--threshold-split-holdout-ratio`
  - `--threshold-split-late-window-ratio`
  - `--threshold-split-holdout-late-fraction`
  - `--rolling-drift-analysis`
  - `--rolling-window-size`
  - `--rolling-step-size`
  - `--rolling-min-train-support`
  - `--rolling-min-holdout-support`

当前项目：

- 参数签名删除 `single_attack_types`
- 不再下发 exploration config / single attack / split / rolling drift 参数
- 固定加入：
  - `--threshold-objective accuracy`
  - `--enable-accuracy-guard <bool>`
  - `--min-failure-detection-accuracy <float>`

### 四、`parse_args(...)` 参数变化

#### 当前项目删除的参数

- `--exploration-config`
- `--single-attack-types`
- `--threshold-objective` 的“临时 override”语义
- `--threshold-min-precision`
- `--threshold-split-mode`
- `--threshold-split-holdout-ratio`
- `--threshold-split-late-window-ratio`
- `--threshold-split-holdout-late-fraction`
- `--rolling-drift-analysis`
- `--rolling-window-size`
- `--rolling-step-size`
- `--rolling-min-train-support`
- `--rolling-min-holdout-support`

#### 当前项目新增的参数

- `--enable-accuracy-guard`
- `--min-failure-detection-accuracy`

#### 保留但简化的参数

- `--coverage-target`
- `--min-samples-for-coverage-stop`
- `--stop-on-coverage-target`

帮助文本与解释都比 `algo-新版0604` 更简化。

### 五、`main(...)` 逻辑变化

#### `algo-新版0604`

- 解析 `exploration_config_path`
- 解析 `single_attack_types`
- 要求 `scenario_exploration.yaml` 文件必须存在
- 组装 `simulation_cmd` 时把 `single_attack_types` 传下去

#### 当前项目

- 删除 `exploration_config_path`
- 删除 `single_attack_types`
- 不再要求 `scenario_exploration.yaml` 存在
- `run_id` 支持优先取 `database_writer.task_id`
- 组装 `simulation_cmd` 时不再传 `single_attack_types`

### 结论

- `run_full_project_pipeline.py` 是总流程入口层面的主分叉点。
- 当前项目更强调“固定链路 + 额外指标记录 + accuracy guard”。
- `algo-新版0604` 更强调“探索参数配置化 + CLI override”。
