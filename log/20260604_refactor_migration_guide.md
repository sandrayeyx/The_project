# 目录重构迁移指南

本文档用于指导旧版本项目向当前目录规范迁移。重点不是记录某一次零散改动，而是沉淀一套可复用的方法：怎么规划、怎么迁、常见问题是什么、应该怎么验证。

## 1. 重构目标

本次目录重构的最终目标是把项目顶层结构收敛为以下几类：

- `src`
- `data`
- `config`
- `log`
- 少量根目录入口脚本

对应约束如下：

- `src`：只放源码、模块私有配置、与代码强绑定的说明文件
- `data`：只放静态资源和运行产物
- `config`：只放项目级公共配置
- `log`：只放日志、方案、执行记录、问题分析文档
- 根目录：只保留确有必要的入口脚本和项目元信息文件

## 2. 目录职责定义

### 2.1 `src`

应迁入 `src` 的内容：

- 业务源码目录
- 测试源码目录
- 代码强依赖的模块私有配置
- 与某个模块紧耦合的说明文档

本次重构中的典型迁移：

- `online_self_healing` -> `src/online_self_healing`
- `iterative_testing` -> `src/iterative_testing`
- `failure_and_attribution_analysis` -> `src/failure_and_attribution_analysis`
- `tests` -> `src/tests`

### 2.2 `data`

`data` 分为两类：

- 静态资源
- 运行产物

静态资源统一放在顶层 `data` 下的专用子目录：

- `data/Ground_Data`
- `data/Satellite_Data`
- `data/ne_data`
- `data/model_weights`
- `data/train_config_archive`

运行产物统一放在：

- `data/tmp_data`

其中典型子目录包括：

- `data/tmp_data/full_project_runs`
- `data/tmp_data/smoke_runs`
- `data/tmp_data/closed_loop_outputs`
- `data/tmp_data/data_archive`
- `data/tmp_data/training_process_data`
- `data/tmp_data/invalid_batch`
- `data/tmp_data/model_artifacts`

### 2.3 `config`

应迁入 `config` 的内容：

- 主流程公共配置
- 环境配置
- pipeline 配置
- runtime 配置

本次重构中的典型迁移：

- `env_config.md` -> `config/environment/env_config.md`

模块私有配置不强行迁出，允许保留在 `src/<module>/...`

### 2.4 `log`

`log` 保持不变，用来沉淀：

- 重构方案
- 问题分析
- 执行记录
- 后续排障记录

## 3. 重构方法

### 3.1 先定规则，再动目录

不要一开始就直接移动文件。正确顺序是：

1. 先定义顶层目录职责
2. 再确认每个旧目录属于 `src/data/config/log` 的哪一类
3. 再建立“旧路径 -> 新路径”映射表
4. 最后才做真实迁移和代码修正

这样做的原因是，项目往往不是“目录乱”，而是“职责混杂”。如果没有先定义职责，移动之后还会继续乱。

### 3.2 先处理路径治理，再处理物理迁移

建议顺序：

1. 先统一运行产物输出路径
2. 再统一静态资源读取路径
3. 再迁移源码目录
4. 再迁移公共配置
5. 最后处理兼容层和历史残留

原因是：

- 运行产物和静态资源路径最容易导致“代码能跑但结果写错地方”
- 源码目录迁移本身不可怕，真正难的是各种硬编码路径

### 3.3 先保留兼容，再逐步收口

第一次重构时，不建议立刻删除所有旧入口。

更稳妥的方式是：

- 先保留少量兼容入口
- 把真实逻辑逐步切到新结构
- 验证主流程跑通后，再决定是否进一步收口

但兼容层不能无限保留，否则会形成“双真身”。后续确认稳定后，应明确谁是最终实现、谁可以删掉。

## 4. 本次重构中的典型问题

### 4.1 源码、静态资源、产物混放

问题表现：

- 模块目录内直接放 `Satellite_Data`
- 模块目录内直接放模型权重
- 分析目录内既有源码又有 `full_project_runs`、`data_archive` 一类运行结果

风险：

- 很难判断哪些是正式资源、哪些是临时结果
- 重构后容易重复保留多套副本
- 代码会持续依赖错误位置

解决方案：

- 静态资源统一提升到顶层 `data`
- 运行产物统一收口到 `data/tmp_data`
- 模块内部保留的重复资源一律视为待淘汰副本

### 4.2 旧路径硬编码

问题表现：

- 代码直接写 `./Ground_Data`
- 代码直接写 `failure_and_attribution_analysis/full_project_runs`
- 子进程脚本直接写旧目录下的 `PRC.py`

本次实际遇到的典型案例：

- `iterative_failure_simulation.py` 仍然调用旧路径 `iterative_testing/PRC.py`
- `run_batch_experiments.py` 也存在同类问题

风险：

- 目录搬了，但流程一跑就断
- 一部分功能表面正常，进入子流程才报错

解决方案：

- 所有入口统一通过路径常量派生
- 子进程启动目标一并改成新位置
- 不只搜顶层入口，还要搜 `subprocess`、`Path(...)`、字符串拼接路径

### 4.3 自己手算项目根目录，导致层级算错

问题表现：

- 某些脚本没有复用统一路径模块
- 而是自己写 `CURRENT_DIR.parents[...]` 推导 `PROJECT_ROOT`

本次实际踩到的问题：

- `run_2_2module_pipeline.py`
- `run_candidate_retraining.py`

这两个脚本内部自己推导项目根时多偏了一层，结果把项目根错误算成：

- `E:\\project\\code`

而不是：

- `E:\\project\\code\\algo`

导致重训练摘要路径、候选模型目录等被拼到错误根下。

解决方案：

- 新结构下统一复用 [project_paths.py](/abs/e:/project/code/algo/src/project_paths.py:1)
- 不允许每个模块再各自手算 `PROJECT_ROOT`
- 如果确实需要启动引导，只允许保留极小的 `sys.path` bootstrap，真正根路径必须来自统一模块

### 4.4 兼容壳和真实实现重复存在

问题表现：

- 根目录有 `run_full_project_pipeline.py`
- `src/tools` 里又有一份同名实现
- 根目录文件只是转发壳

风险：

- 用户分不清哪个才是“真的”
- 后续修改可能只改了一份，另一份继续陈旧
- 与“根目录保留少量入口脚本”的认知相冲突

解决方案：

- 明确用户期望是“根目录直接保留真实入口文件”
- 把真实实现放回根目录
- 删除 `src/tools` 中对应重复文件
- 全项目检查是否还有对旧封装层的引用

### 4.5 旧历史路径残留在数据中

问题表现：

- 训练注册表、历史归档、旧报告中保留了老机器或老版本的绝对路径

风险：

- 分析时容易误以为当前代码仍在使用旧路径
- 新老版本对照时容易判断失真

解决方案：

- 历史数据允许保留旧路径，视为历史记录
- 但不能把历史路径继续当成当前结构依据
- 判断是否修好，必须看“新跑出来的结果”是否落到新目录

## 5. 这次重构的核心解决方案

### 5.1 建立统一路径模块

新增统一路径入口：

- [project_paths.py](/abs/e:/project/code/algo/src/project_paths.py:1)

作用：

- 提供统一的 `PROJECT_ROOT`
- 提供 `SRC_ROOT / DATA_ROOT / TMP_DATA_ROOT / CONFIG_ROOT / LOG_ROOT`
- 提供静态资源路径常量
- 提供运行产物路径常量
- 提供 `ensure_project_paths_on_syspath()`

迁移旧版本时，建议优先建立这一个文件，再开始批量改路径引用。

### 5.2 明确静态资源唯一来源

本次最终确定的正式静态资源来源是：

- `data/Satellite_Data`
- `data/model_weights`

模块内部的：

- `online_self_healing/Satellite_Data`
- `online_self_healing/model_weight`

都视为重复副本，已归档，不再作为正式来源。

旧版本迁移时，也应先做“唯一来源”判定，否则后面会出现“文件改了但实际没生效”的问题。

### 5.3 统一运行产物出口

所有测试结果、中间结果、归档结果统一写入：

- `data/tmp_data`

这样做的好处：

- 代码目录保持干净
- 清理产物时不会误伤源码
- 结果路径规律化，方便后续批量分析

### 5.4 顶层公共配置集中管理

公共环境配置迁入：

- `config/environment/env_config.md`

训练模板与归档型 yaml 则放在：

- `data/train_config_archive`

这两类不要混在一起。前者是“当前主流程配置”，后者更像“样本模板和历史训练配置集合”。

## 6. 推荐给旧版本的执行顺序

建议严格按以下顺序执行：

1. 建立目录职责说明
2. 建立旧路径到新路径映射表
3. 先创建 `src/project_paths.py`
4. 统一运行产物路径到 `data/tmp_data`
5. 统一静态资源路径到 `data/...`
6. 排查所有硬编码路径和子进程调用路径
7. 迁移源码目录到 `src`
8. 迁移公共配置到 `config`
9. 处理重复资源副本和历史残留
10. 再决定兼容壳保留还是删除
11. 最后跑通主流程验证

不建议把“移动目录”和“修路径”拆得太开。目录已经变了但路径没收干净时，最容易出现隐蔽问题。

## 7. 验证方法

目录重构完成后，不要只看文件位置，要做以下验证：

### 7.1 导入验证

检查关键入口和关键模块是否还能导入：

- 主流程入口
- 关键子模块
- 测试模块

### 7.2 子进程验证

重点检查所有 `subprocess` 启动的脚本：

- 是否仍引用旧目录
- 是否仍写到旧输出位置

### 7.3 新运行结果落点验证

至少跑一轮完整流程，检查新结果是否真正落在：

- `data/tmp_data/full_project_runs`
- `data/tmp_data/data_archive`
- `data/tmp_data/model_artifacts`

如果新结果仍跑到旧位置，说明路径治理还没完成。

### 7.4 报告内容验证

不只看文件有没有生成，还要看报告内部记录的路径：

- `pipeline_summary.json`
- `run_manifest.json`
- 重训练摘要报告

因为有些问题不是“没生成”，而是“生成到了错误根目录”。

## 8. 给旧版本迁移时的特别提醒

### 8.1 不要把历史数据一起强行“规范化”

旧版本已有的历史归档、历史报告、历史注册表，可以保留原样。重构重点是：

- 当前代码路径改对
- 新生成结果落对

不是要求把所有旧数据全部改写一遍。

### 8.2 不要默认根目录脚本一定要搬进 `src`

如果项目使用习惯上就是从根目录运行：

- `python run_full_project_pipeline.py`

那根目录入口保留真实实现是合理的。不要为了形式统一，把用户最常用入口封成多层壳。

### 8.3 不要让一个目录同时承担“源码”和“产物”

这是旧版本最容易反复回潮的问题。只要允许运行结果继续写回源码树，目录很快又会重新变乱。

## 9. 本次重构可复用的结论

可以直接复用到旧版本升级中的关键原则如下：

- 先定职责，再迁目录
- 先治路径，再搬代码
- 静态资源必须有唯一正式来源
- 运行产物必须统一写入 `data/tmp_data`
- 公共配置和训练归档要分开
- 统一使用 `project_paths.py`，禁止模块各自手算根路径
- 子进程调用路径必须单独检查
- 历史数据可保留，但新数据必须严格符合新结构
- 根目录入口是否保留，应以实际使用方式为准

## 10. 建议的落地物

旧版本开始迁移前，建议至少准备以下文档或文件：

- 一份目录职责文档
- 一份旧路径 -> 新路径映射表
- 一份问题分析文档
- 一份执行记录文档
- 一个统一路径模块 `src/project_paths.py`

如果这几样先准备好，后续重构会稳很多，也更容易排查“为什么改了目录却还是不对”。
